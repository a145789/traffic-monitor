## 一、 高优先级问题 (潜在 Bug 与资源泄漏)

### 1. 锁屏/唤醒及全屏状态交替导致的「鼠标轮询线程泄漏与抢占」
* **受影响代码**：[main.rs:L313-356](file:///D:/work_space/life/traffic-monitor/src/main.rs#L313-356) 及 [mouse_hid.rs:L81-91](file:///D:/work_space/life/traffic-monitor/src/mouse_hid.rs#L81-91)
* **漏洞成因**：
  在主窗口消息循环 `wnd_proc` 中，系统唤醒事件 `PBT_APMRESUMEAUTOMATIC` 和锁屏解锁事件 `WTS_SESSION_UNLOCK` 被独立接收。当系统从睡眠状态恢复并解锁时，这两个消息通常会在极短时间差内相继到来。
  当收到任一消息时，程序会直接执行：
  ```rust
  MOUSE_THREAD = Some(start_mouse_thread());
  ```
  如果上一个线程的 `JoinHandle` 仍在 `MOUSE_THREAD` 中且没有被 `stop_mouse_thread()` + `join()` 显式等待销毁，直接将其赋予 `Some` 会使上一个线程的 `JoinHandle` 被直接 **Drop（丢弃）**。
  由于 Rust 中丢弃 `JoinHandle` 仅表示将其从主线程**分离（Detach）**，该子线程仍会在后台继续循环运行。且新线程启动时，全局 `SHOULD_STOP` 标志又会被重新设置为 `false`。
* **潜在后果**：
  多次睡眠唤醒或锁屏解锁后，后台将驻留多个无主（Detached）的 `mouse_worker_loop` 线程。它们会互相竞争独占的鼠标 HID 设备、导致 HID 读取数据错乱、轮询频繁失败，同时引起持续的进程级线程与内存泄漏。
* **修复建议**：
  在启动新线程之前，必须检查 `MOUSE_THREAD`，如果存在则先执行停止与 `join` 回收；或者在解锁/唤醒时判断，若当前线程已处于运行态则不再重复启动。

### 2. HidApi 每次循环重建导致的性能浪费
* **受影响代码**：[mouse_hid.rs:L115-121](file:///D:/work_space/life/traffic-monitor/src/mouse_hid.rs#L115-121)
* **问题描述**：
  `mouse_worker_loop()` 每次迭代都会创建新的 `HidApi` 实例：
  ```rust
  let api = match HidApi::new() {  // 每次循环都重建
      Ok(api) => api,
      Err(_) => { ... }
  };
  ```
  `HidApi::new()` 初始化整个底层 HID 子系统，涉及较重的系统调用。在鼠标离线时（每 5 分钟）和在线时（每 3 分钟），频繁地重建并销毁该实例会造成不必要的开销与潜在的驱动级不稳定。
* **修复建议**：
  将 `HidApi` 实例缓存到循环体之外，仅在检测到 API 状态异常或需要重建时才重新创建。

### 3. 开机自启绝对路径含空格导致启动失败
* **受影响代码**：[tray.rs:L231-244](file:///D:/work_space/life/traffic-monitor/src/tray.rs#L231-244)
* **漏洞成因**：
  程序通过读取 `std::env::current_exe()` 并调用 `RegSetValueExW` 将绝对路径直接写入注册表 `Run` 键中。
  当用户的程序存放目录绝对路径中包含空格（例如 `D:\Program Files\TrafficMonitor\traffic-monitor.exe`）时，Windows 启动加载器在解析命令时，会因为路径没有被双引号包裹而被空格截断，错误地尝试运行 `D:\Program` 并传入后续的参数。
* **潜在后果**：
  若用户把软件放在带空格的文件夹下，点击“开机自启”虽然提示成功，但在系统下次重启时不会自启。
* **修复建议**：
  在写入注册表路径时，使用双引号包裹绝对路径：
  ```rust
  let path_str = exe_path.to_string_lossy().to_string();
  let path_quoted = format!("\"{}\"", path_str);
  ```

## 二、 中优先级问题 (内存安全隐患与体验 Bug)

### 4. 指针强制类型转换导致的「内存未对齐」未定义行为 (UB)
* **受影响代码**：[collector.rs:L84-85](file:///D:/work_space/life/traffic-monitor/src/collector.rs#L84-L85)
* **具体问题**：
  在 `collect_network` 函数中，程序申请了一个以字节（`u8`）为单位的缓冲区，并强制将其指针转型为 `MIB_IFTABLE` 类型：
  ```rust
  let mut buf = vec![0u8; buf_size as usize];
  let table = buf.as_mut_ptr() as *mut MIB_IFTABLE;
  ```
  在 Rust 中，`Vec<u8>` 的内存对齐保证是 `1` 字节。而 `MIB_IFTABLE` 属于具有特定高对齐要求的 C 结构体（一般需要 `4` 字节或 `8` 字节对齐）。
  直接将低对齐要求的指针转换为高对齐要求的指针，并在 Rust 代码中通过 `let table_ref = &*table;` 进行解引用，违反了 Rust 的内存安全契约，属于 **未定义行为 (Undefined Behavior, UB)**。
* **潜在后果**：
  在高度优化的 Release 模式下，未对齐的解引用可能导致未知的编译器优化 Bug、硬件级读取对齐异常或返回错乱的网卡状态。
* **修复建议**：
  改用 `Vec<u32>` 或 `Vec<u64>` 作为内存分配的载体（自动具备 4 字节 / 8 字节对齐），或直接重构使用 `GetIfTable2` 并让系统接口分配正确对齐的缓冲区。

### 5. 32 位网卡流量数据溢出导致网速周期性归零
* **受影响代码**：[collector.rs:L107-123](file:///D:/work_space/life/traffic-monitor/src/collector.rs#L107-L123)
* **具体问题**：
  当前的流量采样采用 `GetIfTable`，其返回的网卡字节统计字段 `dwInOctets` 和 `dwOutOctets` 均为 32 位无符号整数（上限约 4.29 GB）。
  一旦用户的宽带速度较快（例如千兆网卡，下载速率 125 MB/s），仅需 30~40 秒的时间就会触发 32 位流量溢出。
  溢出发生时，`total_in` 会突变回一个极小值，从而由于 `total_in < PREV_NET_IN` 走向以下降级逻辑：
  ```rust
  if total_in >= PREV_NET_IN {
      // ...
  } else {
      NET_SPEED_DOWN.store(0, Ordering::Relaxed); // 溢出时强制归零
  }
  ```
* **潜在后果**：
  在高速下载文件或大吞吐量传输时，界面显示的网速会频繁且规律地“抽风”归零（显示 `0 B/s`），直到下一次采样。
* **优化建议**：
  **迁移至 `GetIfTable2` API**。其对应的 `MIB_IF_ROW2` 结构体提供 64 位无符号整数网速指标（`InOctets`、`OutOctets` 等为 `u64` 规格），能永久避免溢出归零现象，并能原生地基于接口连接标志过滤出真实的物理网卡，精简网卡筛选逻辑。

### 6. `collect_network` 缓冲区大小竞态缺陷
* **受影响代码**：[collector.rs:L77-87](file:///D:/work_space/life/traffic-monitor/src/collector.rs#L77-L87)
* **具体问题**：
  在两次 `GetIfTable` 调用之间，网络接口数量若发生改变，可能会触发缓冲区不足错误：
  ```rust
  GetIfTable(None, &mut buf_size, false);  // 第一次：获取所需大小
  let mut buf = vec![0u8; buf_size as usize];
  let table = buf.as_mut_ptr() as *mut MIB_IFTABLE;
  let result = GetIfTable(Some(table), &mut buf_size, false);  // 第二次：若此时网卡增多，会报错返回并使本次网速采集失败
  ```
* **潜在后果**：在插拔网线或虚拟网卡频繁变动时，网速统计可能会有极短暂的失效。
* **修复建议**：增加重试逻辑，或者在迁移至 `GetIfTable2` 时，直接借助系统分配的缓冲区（`GetIfTable2` 会在内部自动分配正确大小的对齐内存）。

## 三、 低优先级与代码质量问题 (DPI 健壮性与代码异味)

### 7. 屏幕 DPI 切换时 Compatible Bitmap 的有效性缺失
* **受影响代码**：[renderer.rs:L194-206](file:///D:/work_space/life/traffic-monitor/src/renderer.rs#L194-L206)
* **具体问题**：
  在 `update_dpi` 方法中，程序在 DPI 改变时会通过 `CreateCompatibleBitmap` 分配新的位图。
  如果系统在频繁改变 DPI 或硬件高负载时资源紧张，分配可能失败并返回空句柄 `HBITMAP(null)`。
  当前代码并未对其进行合法性检测，若直接执行 `SelectObject(self.hdc_mem, new_bitmap)`，会导致设备上下文处于非法状态，且随后会将原本正常并被换出来的旧 `self.hbitmap` 销毁。
* **优化建议**：
  在 `SelectObject` 之前对新创建的位图和字体进行有效性校验（如果不为 null 再执行替换和删除旧位图的操作）。

### 8. 主程序退出时静态全局 Renderer 未释放
* **受影响代码**：[main.rs:L40](file:///D:/work_space/life/traffic-monitor/src/main.rs#L40)
* **具体问题**：
  `RENDERER` 声明为静态变量 `static mut RENDERER: Option<Renderer>`。在 `main` 循环因为 `WM_CLOSE` 退出并结束进程时，并没有显式执行 `RENDERER = None` 或 `.take()`。
  静态局部变量在进程销毁时不会被自动调用 `drop`，导致 `Renderer` 的析构函数未运行，虽然 Windows 会对死亡的进程进行强制句柄清空，但这对严苛的内存泄漏检测工具仍会抛出 GDI 资源未释放警告。
* **优化建议**：
  在退出 `main` 函数前，调用 `let _ = RENDERER.take();` 显式析构内存 DC、位图和字体资源。

### 9. 托盘图标未设置版本号
* **受影响代码**：[tray.rs:L78-93](file:///D:/work_space/life/traffic-monitor/src/tray.rs#L78-L93)
* **具体问题**：
  创建托盘图标时未显式设置 `uVersion`。导致可能无法顺畅利用 Windows 10/11 的高级通知或不同版本的通知响应特性。
* **修复建议**：
  在创建 `NOTIFYICONDATAW` 时显式设置 `uVersion = NOTIFYICON_VERSION_4`，并在 `Shell_NotifyIconW(NIM_ADD, &nid)` 后调用一次 `NIM_SETVERSION` 指令。

### 10. `static mut` 变量跨线程访问的理论风险
* **受影响代码**：[main.rs:L40-44](file:///D:/work_space/life/traffic-monitor/src/main.rs#L40-L44)
* **具体问题**：
  虽然当前代码的线程安全逻辑是健全的（例如鼠标线程通过原子变量与消息机制通信），但 Rust 编译器对跨生命周期操作 `static mut` 仍保持极高警惕，未来编译器优化可能会在此处产生理论风险。
* **修复建议**：
  长期来看，可以考虑用更加线程安全的机制封装或保持仅在单线程（主线程）中访问。

### 11. DPI 变化时重复更新主题颜色
* **受影响代码**：[main.rs:L305-311](file:///D:/work_space/life/traffic-monitor/src/main.rs#L305-L311)
* **具体问题**：
  在 `WM_DPICHANGED` 中，同时调用了 `update_dpi()` 与 `embed_in_taskbar()`，由于重新嵌入任务栏会触发 `WM_SETTINGCHANGE`，从而间接导致 `update_text_color()` 被重复触发，增加了微小的无意义开销（重复读取注册表）。
* **修复建议**：
  移除 `WM_DPICHANGED` 分支中直接更新 DPI 时多余的色彩更新，交由 `WM_SETTINGCHANGE` 统一分发。

### 12. 代码异味：魔数与过于宽松的错误处理
* **受影响代码**：多处，例如 [renderer.rs:L155](file:///D:/work_space/life/traffic-monitor/src/renderer.rs#L155) 中硬编码的 `122.0`（代表低电量文字定位的 X 轴偏移量）。
* **具体问题**：
  * 魔数 `122.0` 缺乏注释和命名，影响代码的可读性，可以考虑在 `config.rs` 中定义为 `LOW_BATTERY_TEXT_X`。
  * 项目中存在大量 `let _ = ...` 强行忽略系统 API 调用的错误（如 `SetTimer`、`InvalidateRect`、`SelectObject`）。虽然对很多底层 GDI 和窗口指令的失败并无严重影响，但对关键状态应至少保留日志。
  * `main` 函数中的大部分初始化和生命周期代码被包裹在了一个过于庞大的 `unsafe` 块中，应当拆分为更小的安全函数以提高可维护性。
